# CLR (Common Language Runtime) – Interview Q&A

---

## 1. What is the CLR, and what are its primary responsibilities?

**Answer:**

The **Common Language Runtime (CLR)** is the execution engine of .NET. It runs .NET applications and provides core runtime services.

### Key Responsibilities

* **Memory Management** → Automatic garbage collection
* **Execution Management** → Executes compiled code
* **Exception Handling** → Structured error handling
* **Security** → Code verification and access control
* **Thread Management** → Handles multithreading
* **JIT Compilation** → Converts IL to machine code at runtime

**Interview One-liner:**

> CLR is the runtime engine of .NET that manages execution, memory, and system-level services.

---

## 2. Explain the execution process of a .NET application from source code to machine code

**Answer:**

The execution happens in two main stages: compilation to IL and JIT compilation to machine code.

### Step-by-step Process

1. **Write Source Code**

   * Code is written in C# (`.cs` files)

2. **Compile to Intermediate Language (IL)**

   * Using the C# compiler (`csc`)
   * Output: `.exe` or `.dll`
   * Contains IL code and metadata

3. **CLR Loads the Assembly**

   * Verifies code safety
   * Loads dependencies

4. **JIT Compilation**

   * Converts IL into native machine code
   * Happens at runtime (method-by-method)

5. **Execution**

   * Native code runs on the CPU

**Interview One-liner:**

> C# code is compiled into IL, and then CLR uses JIT to convert IL into machine code at runtime.

---

## 3. What is Managed vs Unmanaged Code?

**Answer:**

### Managed Code

* Runs under the control of CLR
* Provides:

  * Garbage collection
  * Exception handling
  * Security enforcement
* Examples: C#, VB.NET

### Unmanaged Code

* Runs outside the CLR
* No automatic memory management
* Developer handles memory manually
* Examples: C++, Win32 APIs

**Interview One-liner:**

> Managed code runs under CLR with automatic services, while unmanaged code runs directly on the OS without CLR support.

---

## 4. What is the role of Common Intermediate Language (CIL/MSIL)?

**Answer:**

**CIL (Common Intermediate Language)**, also called MSIL, is a platform-independent code generated after compiling .NET source code.

### Role

* Acts as a bridge between source code and machine code
* Enables platform independence
* Supports multiple .NET languages

### Execution Flow

```
C# → CIL → JIT → Machine Code
```

**Interview One-liner:**

> CIL is an intermediate, platform-independent code that CLR converts into machine code using JIT.

# CTS (Common Type System) & CLS (Common Language Specification) – Interview Q&A

---

## 1. What is the difference between CTS and CLS?

**Answer:**

### CTS (Common Type System)

* Defines **all data types and programming constructs** in .NET
* Ensures that all .NET languages follow a **common type framework**
* Includes:

  * Value types (int, struct, etc.)
  * Reference types (class, object, etc.)

### CLS (Common Language Specification)

* A **subset of CTS**
* Defines a set of **rules that all .NET languages must follow** to ensure interoperability
* Not all CTS features are included in CLS

### Key Difference

* **CTS** → Defines *all* types and rules
* **CLS** → Defines *only the rules needed for cross-language compatibility*

**Interview One-liner:**

> CTS defines all types in .NET, while CLS is a subset that ensures different languages can work together.

---

## 2. How does the CTS ensure type safety across different .NET languages?

**Answer:**

CTS enforces a **common definition of types and behavior** across all .NET languages.

### How it ensures type safety:

* All languages compile to **Intermediate Language (IL)** using CTS rules
* Defines strict rules for:

  * Type declaration
  * Type usage
  * Type conversion
* Prevents invalid operations (e.g., assigning string to int)
* Ensures objects behave consistently across languages

### Example:

* A class written in C# can be safely used in VB.NET because both follow CTS

**Interview One-liner:**

> CTS ensures type safety by enforcing a common type definition and rules that all .NET languages must follow.

---

## 3. Why is the CLS important for language interoperability?

**Answer:**

CLS ensures that code written in one .NET language can be **used seamlessly in another language**.

### Why it is important:

* Defines a **common set of rules** that all languages agree on
* Avoids language-specific features that others cannot understand
* Ensures:

  * Method signatures are compatible
  * Data types are universally supported
* Promotes **cross-language integration**

### Example:

* If a C# class follows CLS rules, it can be used in VB.NET or F# without issues

### Practical Note:

* Developers can mark code as:

  ```csharp
  [CLSCompliant(true)]
  ```

  to ensure it follows CLS rules

**Interview One-liner:**

> CLS ensures interoperability by enforcing a common set of rules that all .NET languages can understand and follow.

# Garbage Collection (GC) – Interview Q&A

---

## 1. How does the Garbage Collector work in .NET?

**Answer:**

The **Garbage Collector (GC)** in .NET automatically manages memory by reclaiming unused objects.

### How it works:

1. **Allocation**

   * Objects are created on the managed heap

2. **Tracking**

   * GC tracks object references (roots like stack, static variables)

3. **Mark Phase**

   * Identifies objects that are still in use (reachable)

4. **Sweep Phase**

   * Removes objects that are no longer referenced (unreachable)

5. **Compaction**

   * Moves remaining objects together to reduce fragmentation
   * Updates references

### Key Points:

* Runs automatically (non-deterministic)
* Improves memory efficiency
* Prevents memory leaks (to an extent)

**Interview One-liner:**

> GC automatically frees memory by removing unreachable objects and compacting the heap.

---

## 2. Explain the concept of Generations (0, 1, and 2). Why are they used?

**Answer:**

The GC uses a **generational model** to optimize performance based on object lifespan.

### Generations:

* **Generation 0 (Gen 0)**

  * Newly created objects
  * Collected frequently
  * Short-lived objects

* **Generation 1 (Gen 1)**

  * Intermediate objects
  * Acts as a buffer between Gen 0 and Gen 2

* **Generation 2 (Gen 2)**

  * Long-lived objects
  * Collected less frequently
  * Examples: static data, application-level objects

### Why Generations?

* Based on the principle:

  > “Most objects die young”

* Improves performance by:

  * Avoiding full heap scans
  * Focusing on recently created objects

**Interview One-liner:**

> Generations optimize GC by collecting short-lived objects more frequently and long-lived objects less often.

---

## 3. What is the difference between GC.Dispose() and GC.Collect()?

**Answer:**

### GC.Collect()

* Forces garbage collection manually
* Cleans up **managed memory**
* Generally **not recommended** (can hurt performance)

### Dispose() (via IDisposable)

* Used to release **unmanaged resources**
* Called explicitly by developer
* Examples:

  * File handles
  * Database connections

### Key Difference:

* `GC.Collect()` → Handles managed memory (automatic, forced manually)
* `Dispose()` → Releases unmanaged resources (manual cleanup)

**Interview One-liner:**

> GC.Collect forces garbage collection, while Dispose is used to release unmanaged resources explicitly.

---

## 4. What is the Large Object Heap (LOH) and how is it handled?

**Answer:**

The **Large Object Heap (LOH)** stores large objects (typically **> 85 KB**).

### Characteristics:

* Used for large arrays, large strings, etc.
* Allocated separately from the small object heap

### Handling:

* Collected during **Gen 2 collections**
* **Not compacted frequently** (to avoid performance cost)
* Can lead to **memory fragmentation**

### Key Point:

* Frequent large allocations can impact performance

**Interview One-liner:**

> LOH stores large objects and is collected with Gen 2, but is not compacted often to avoid performance overhead.

---

## 5. Explain the purpose of a Finalizer and the IDisposable interface

**Answer:**

### Finalizer (~Destructor)

* Called by GC before object is collected
* Used to clean up **unmanaged resources**
* Syntax:

  ```csharp
  ~MyClass() { }
  ```
* Non-deterministic (you don’t know when it runs)

---

### IDisposable Interface

* Provides a **deterministic way** to release resources
* Method:

  ```csharp
  void Dispose();
  ```

### Best Practice:

* Use `Dispose()` for cleanup
* Use `using` statement:

  ```csharp
  using (var obj = new MyClass()) { }
  ```

### Relationship:

* Finalizer is a **backup mechanism**
* Dispose is the **preferred approach**

---

**Interview One-liner:**

> Dispose provides deterministic cleanup of resources, while finalizers act as a fallback handled by the GC.

# Assemblies, DLLs, and EXEs – Interview Q&A

---

## 1. What is a .NET Assembly, and what does it contain (Metadata, Manifest, etc.)?

**Answer:**

A **.NET Assembly** is the **basic building block of a .NET application**. It is a compiled unit that contains **IL code + metadata + resources**, and is executed by the CLR.

### What an Assembly contains:

#### 1. MSIL (Intermediate Language Code)

* Platform-independent code generated after compilation
* Executed by CLR after JIT compilation

#### 2. Metadata

* Describes the types inside the assembly
* Includes:

  * Classes
  * Methods
  * Properties
  * References to other assemblies

#### 3. Manifest

* Acts as the **identity of the assembly**
* Contains:

  * Assembly name
  * Version
  * Culture
  * Security permissions
  * List of referenced assemblies

#### 4. Resources

* Images, strings, configuration files, etc.

**Interview One-liner:**

> A .NET assembly is a compiled unit containing IL code, metadata, manifest, and resources executed by the CLR.

---

## 2. What is the difference between a Private and a Shared assembly?

**Answer:**

### Private Assembly

* Used by a **single application**
* Stored in the application’s **bin folder**
* No version conflicts
* Simple deployment

### Shared Assembly

* Used by **multiple applications**
* Must have a **strong name (unique identity)**
* Stored in the **Global Assembly Cache (GAC)** (older .NET)
* Requires version management

### Key Difference:

| Feature    | Private Assembly | Shared Assembly |
| ---------- | ---------------- | --------------- |
| Scope      | Single app       | Multiple apps   |
| Location   | App folder       | GAC             |
| Versioning | Simple           | Strict          |
| Sharing    | No               | Yes             |

**Interview One-liner:**

> Private assemblies are app-specific, while shared assemblies are reused across applications with strong naming.

---

## 3. What is the Global Assembly Cache (GAC), and is it still relevant in .NET Core/5+?

**Answer:**

### GAC (Global Assembly Cache):

* A **machine-wide repository** for shared .NET Framework assemblies
* Stores **strong-named assemblies**
* Enables multiple applications to share the same assembly version

### Is it still relevant?

* ❌ **Not used in .NET Core / .NET 5+**
* It was primarily used in **.NET Framework**
* Modern .NET uses:

  * **NuGet package management**
  * **App-local dependencies (self-contained deployment)**

### Why GAC is deprecated in modern .NET:

* Avoids “DLL Hell”
* Encourages side-by-side versioning per application

**Interview One-liner:**

> GAC was used in .NET Framework for shared assemblies but is no longer used in .NET Core and later versions.

---

## 4. Explain the difference between a .dll and a .exe in .NET

**Answer:**

Both `.dll` and `.exe` are **types of assemblies**, but they differ in execution behavior.

### .EXE (Executable)

* Entry point exists (`Main` method)
* Can be **executed directly**
* Represents a standalone application

### .DLL (Dynamic Link Library)

* No entry point
* Cannot run independently
* Used as a **reusable library**

### Key Difference:

| Feature     | .EXE               | .DLL                 |
| ----------- | ------------------ | -------------------- |
| Execution   | Directly runnable  | Not executable       |
| Entry Point | Yes (Main method)  | No                   |
| Purpose     | Application        | Library              |
| Usage       | Standalone program | Shared functionality |

**Interview One-liner:**

> EXE is an executable application with an entry point, while DLL is a reusable library without an entry point.

# JIT Compilation – Interview Q&A

---

## 1. What is JIT (Just-In-Time) compilation?

**Answer:**

**JIT (Just-In-Time) compilation** is a process in .NET where **IL (Intermediate Language) code is converted into native machine code at runtime** by the CLR.

### How it works:

1. C# code is compiled into **IL**
2. CLR loads the assembly
3. JIT compiler converts IL → machine code
4. Machine code runs on the CPU

### Key idea:

* Compilation happens **just before execution (not ahead of time)**

**Interview One-liner:**

> JIT compiles IL into native machine code at runtime when a method is first called.

---

## 2. Compare Normal JIT, Econo-JIT, and Pre-JIT (AOT)

**Answer:**

### 🔹 Normal JIT (Standard JIT)

* Compiles methods **on demand (first execution)**
* Stores compiled code for reuse during runtime
* Most common in modern .NET

### 🔹 Econo-JIT (Economy JIT)

* Used in older/limited environments (historical .NET)
* Compiles methods when needed but may **discard compiled code to save memory**
* Less common today

### 🔹 Pre-JIT (AOT – Ahead-of-Time Compilation)

* Entire application is compiled **before execution**
* Done using tools like:

  * NGen (Native Image Generator) in .NET Framework
  * ReadyToRun / NativeAOT in modern .NET

### Comparison Table:

| Type          | When Compilation Happens   | Performance  | Usage                            |
| ------------- | -------------------------- | ------------ | -------------------------------- |
| Normal JIT    | Runtime (on demand)        | Balanced     | Default in .NET                  |
| Econo-JIT     | Runtime (memory optimized) | Lower        | Rare/legacy                      |
| Pre-JIT (AOT) | Before runtime             | Fast startup | NativeAOT, mobile, microservices |

**Interview One-liner:**

> Normal JIT compiles on demand, Econo-JIT optimizes memory usage, and Pre-JIT compiles everything ahead of time for faster startup.

---

## 3. How does JIT compilation impact startup performance of an application?

**Answer:**

JIT compilation affects startup because **methods are compiled only when they are first executed**.

### Impact on startup:

#### ❌ Negative impact:

* Initial delay when application starts
* First-time method calls are slower
* Cold start latency increases

#### ✔ Positive impact:

* Only used methods are compiled (no wasted compilation)
* Optimized for actual runtime usage
* Reduces memory usage compared to full precompilation

### Example:

* First API request in a web app may be slower
* Subsequent requests are faster because methods are already JIT-compiled

### Optimization techniques:

* **PreJIT / ReadyToRun images**
* **AOT compilation (.NET Native / NativeAOT)**
* Reducing startup-heavy code paths

**Interview One-liner:**

> JIT improves runtime efficiency but can slow down application startup due to on-demand method compilation.

# Value Types vs Reference Types – Interview Q&A

---

## 1. What is the fundamental difference between a value type and a reference type?

**Answer:**

The key difference is **how data is stored and copied in memory**.

### 🔹 Value Types

* Store the **actual data directly**
* When assigned, a **copy of the value is created**
* Each variable has its own copy

**Examples:**

* `int`, `float`, `bool`
* `struct`, `enum`

---

### 🔹 Reference Types

* Store a **reference (memory address)** to the actual data
* Multiple variables can point to the same object
* Changes affect all references

**Examples:**

* `class`, `string`, arrays, delegates

---

### 🔑 Key Difference:

> Value types copy data, reference types copy memory references.

---

## 2. Explain "Stack" vs "Heap" memory allocation

**Answer:**

### 🔹 Stack Memory

* Stores **value types and method call data**
* Fast allocation and deallocation (LIFO structure)
* Automatically managed
* Each thread has its own stack

**Used for:**

* Local variables
* Method parameters (value types)

---

### 🔹 Heap Memory

* Stores **reference type objects**
* Managed by **Garbage Collector (GC)**
* Slower than stack but flexible
* Shared across the application

**Used for:**

* Objects created using `new`
* Reference type data

---

### 🔑 Key Difference:

| Feature    | Stack       | Heap            |
| ---------- | ----------- | --------------- |
| Speed      | Fast        | Slower          |
| Size       | Limited     | Large           |
| Management | Automatic   | GC-managed      |
| Data type  | Value types | Reference types |

---

## 3. What are Boxing and Unboxing? Why should they be avoided in performance-critical code?

**Answer:**

### 🔹 Boxing

* Converting a **value type → reference type**
* Value is wrapped inside an object and stored in heap

```csharp
int x = 10;
object obj = x; // Boxing
```

---

### 🔹 Unboxing

* Converting **reference type → value type**
* Requires explicit casting

```csharp
int y = (int)obj; // Unboxing
```

---

### ❌ Why avoid Boxing/Unboxing?

* Causes **extra memory allocation on heap**
* Triggers **Garbage Collection overhead**
* Reduces performance in loops and high-frequency operations

---

### 🔑 Interview One-liner:

> Boxing copies a value type to the heap as an object, and unboxing extracts it back, both causing performance overhead.

---

## 4. Is a string a value type or a reference type? Why does it behave like a value type?

**Answer:**

### 🔹 String is a Reference Type

* `string` is a class in .NET → so it is a **reference type**
* Stored on the heap

---

### 🔹 Why it behaves like a value type:

Because **strings are immutable**.

### What does immutability mean?

* Once a string is created, it **cannot be modified**
* Any modification creates a **new string object**

---

### Example:

```csharp
string a = "hello";
string b = a;

b = "world";
```

* `a` remains `"hello"`
* `b` becomes `"world"`
* No shared mutation occurs

---

### 🔑 Why this design?

* Ensures **safety and predictability**
* Enables **string interning optimization**
* Makes strings behave like value types in usage

---

### 🔑 Interview One-liner:

> String is a reference type, but behaves like a value type because it is immutable, meaning every modification creates a new object instead of changing the existing one.

# OOP Concepts – Interview Q&A (C#)

---

## 1. How do you implement Encapsulation in C#?

**Answer:**

**Encapsulation** is the process of **hiding internal data and exposing only necessary functionality**.

### How to implement it in C#:

* Use **private fields**
* Provide controlled access using:

  * Properties (`get` / `set`)
  * Methods

### Example:

```csharp id="enc01"
class BankAccount
{
    private double balance; // hidden data

    public double Balance
    {
        get { return balance; }
        private set { balance = value; }
    }

    public void Deposit(double amount)
    {
        if (amount > 0)
            balance += amount;
    }
}
```

### Key Idea:

* Data is protected from direct modification

**Interview One-liner:**

> Encapsulation hides internal data using private fields and exposes controlled access via properties or methods.

---

## 2. Difference between Method Overloading and Method Overriding

**Answer:**

### 🔹 Method Overloading (Compile-time Polymorphism)

* Same method name
* Different parameter list
* Happens in the **same class**
* Decided at **compile time**

```csharp id="ovl01"
class MathOps
{
    public int Add(int a, int b) => a + b;
    public double Add(double a, double b) => a + b;
}
```

---

### 🔹 Method Overriding (Runtime Polymorphism)

* Same method signature
* Occurs in **base and derived classes**
* Requires `virtual` and `override`

```csharp id="ovr01"
class Animal
{
    public virtual void Speak() => Console.WriteLine("Animal sound");
}

class Dog : Animal
{
    public override void Speak() => Console.WriteLine("Bark");
}
```

---

### Key Difference:

| Feature           | Overloading  | Overriding       |
| ----------------- | ------------ | ---------------- |
| Polymorphism type | Compile-time | Runtime          |
| Method signature  | Different    | Same             |
| Inheritance       | Not required | Required         |
| Keyword           | None         | virtual/override |

**Interview One-liner:**

> Overloading is same method name with different parameters (compile-time), while overriding is redefining a base class method in derived class (runtime).

---

## 3. Explain the "Composition over Inheritance" principle

**Answer:**

This principle suggests:

> “Prefer **using objects (composition)** instead of inheriting from classes.”

### Why?

* Inheritance creates tight coupling
* Composition provides flexibility and reusability

---

### ❌ Inheritance approach:

```csharp id="inh01"
class Engine {}

class Car : Engine {}
```

* Car *is not* an Engine → bad design

---

### ✔ Composition approach:

```csharp id="comp01"
class Engine
{
    public void Start() => Console.WriteLine("Engine started");
}

class Car
{
    private Engine engine = new Engine();

    public void StartCar()
    {
        engine.Start();
    }
}
```

---

### Benefits:

* Loose coupling
* Easier testing
* More flexible design

**Interview One-liner:**

> Composition over inheritance means building classes using other objects instead of relying on rigid inheritance hierarchies.

---

## 4. How does Polymorphism work at runtime vs compile time?

**Answer:**

Polymorphism means **“many forms”**, and it works in two ways:

---

## 🔹 Compile-Time Polymorphism (Static Binding)

* Achieved via **method overloading**
* Method call resolved at compile time
* Faster execution

### Example:

```csharp id="pt01"
Add(int, int)
Add(double, double)
```

✔ Decision made by compiler

---

## 🔹 Runtime Polymorphism (Dynamic Binding)

* Achieved via **method overriding**
* Uses **virtual/override methods**
* Method resolved at runtime based on object type

### Example:

```csharp id="pt02"
Animal obj = new Dog();
obj.Speak(); // Dog's method called
```

✔ Decision made during execution

---

### Key Difference:

| Feature     | Compile-Time  | Runtime         |
| ----------- | ------------- | --------------- |
| Binding     | Early binding | Late binding    |
| Mechanism   | Overloading   | Overriding      |
| Performance | Faster        | Slightly slower |
| Flexibility | Low           | High            |

---

**Interview One-liner:**

> Compile-time polymorphism is method overloading resolved by compiler, while runtime polymorphism is method overriding resolved during execution.

# Interfaces vs Abstract Classes – Interview Q&A

---

## 1. When would you use an Abstract Class instead of an Interface?

**Answer:**

You use an **Abstract Class** when you want to provide:

* **Common base functionality**
* **Shared state (fields)**
* A partially implemented base design

### Use Abstract Class when:

* You want to share **code among related classes**
* You need **constructor logic**
* You want **protected members**
* Classes are **closely related (is-a relationship)**

### Example:

```csharp id="abs01"
abstract class Animal
{
    public string Name;

    public Animal(string name)
    {
        Name = name;
    }

    public void Eat()
    {
        Console.WriteLine("Eating...");
    }

    public abstract void Speak();
}
```

---

### Use Interface when:

* You only want to define a **contract**
* Unrelated classes must follow the same behavior
* You need **multiple inheritance support**

```csharp id="int01"
interface IFlyable
{
    void Fly();
}
```

---

### Key Difference:

> Abstract class is for shared base behavior, while interface is for defining a contract.

---

## 2. Can an Interface have a constructor? What about method bodies (default implementations)?

**Answer:**

### ❌ Can an interface have a constructor?

No.

* Interfaces cannot be instantiated
* So they cannot have constructors

---

### ✔ Can interfaces have method bodies?

Yes (since **C# 8.0+**).

This is called **Default Interface Implementation**.

### Example:

```csharp id="defint01"
interface ILogger
{
    void Log(string message);

    void LogError(string message)
    {
        Console.WriteLine("Error: " + message);
    }
}
```

---

### Why was this added?

* To evolve interfaces without breaking existing implementations
* To support backward compatibility

---

### Key Rule:

* Abstract class → full implementation allowed
* Interface → contract + optional default behavior (modern C#)

---

## 3. Can a class inherit from multiple abstract classes? Why or why not?

**Answer:**

### ❌ No, a class cannot inherit from multiple abstract classes in C#.

### Why?

C# does NOT support multiple inheritance of classes due to:

* **Ambiguity problem (Diamond Problem)**
* Complexity in method resolution
* Maintenance issues

---

### Example of problem:

If two base classes have the same method:

```csharp
class A { public void Show() {} }
class B { public void Show() {} }

// class C : A, B ❌ Not allowed
```

The compiler would not know which `Show()` to use.

---

### ✔ What C# allows instead:

* Single class inheritance
* Multiple interface inheritance

```csharp id="multi01"
class C : A, IFlyable, IPrintable
{
}
```

---

### Key Difference:

> C# allows multiple interfaces but only single class inheritance to avoid ambiguity and complexity.

---

### Interview One-liner:

> A class cannot inherit multiple abstract classes in C# due to the diamond problem, but it can implement multiple interfaces.

# Delegates, Events, and Lambdas – Interview Q&A (C#)

---

## 1. What is a Delegate, and how does it differ from a regular function call?

**Answer:**

A **delegate** is a **type-safe reference to a method**. It allows methods to be passed as parameters or stored in variables.

### Key idea:

* Regular function call → directly invokes a method
* Delegate → stores a reference to a method and invokes it indirectly

---

### Example:

```csharp id="d01"
delegate void MyDelegate(string message);

class Program
{
    static void Print(string msg)
    {
        Console.WriteLine(msg);
    }

    static void Main()
    {
        MyDelegate del = Print;
        del("Hello via delegate");
    }
}
```

---

### Why delegates are useful:

* Callback mechanisms
* Event handling
* Functional programming style

---

### Interview One-liner:

> A delegate is a type-safe function pointer that allows methods to be passed and invoked indirectly.

---

## 2. Difference between Action, Func, and Predicate

**Answer:**

These are **built-in generic delegates** in .NET.

---

### 🔹 Action

* Returns **void**
* Can take 0 to 16 parameters

```csharp id="a01"
Action<string> log = msg => Console.WriteLine(msg);
```

---

### 🔹 Func

* Returns a **value**
* Last generic type is return type

```csharp id="f01"
Func<int, int, int> add = (a, b) => a + b;
```

---

### 🔹 Predicate

* Always returns **bool**
* Used for conditions

```csharp id="p01"
Predicate<int> isEven = x => x % 2 == 0;
```

---

### Key Difference:

| Type      | Return Type | Use Case           |
| --------- | ----------- | ------------------ |
| Action    | void        | Perform operations |
| Func      | Any type    | Return value logic |
| Predicate | bool        | Conditions/filters |

---

### Interview One-liner:

> Action performs operations, Func returns a value, and Predicate returns a boolean condition.

---

## 3. What is the relationship between a Delegate and an Event?

**Answer:**

An **event is built on top of a delegate**.

### Relationship:

* Delegate → defines method signature
* Event → restricts delegate usage to **publish/subscribe model**

```csharp id="i01"
IEnumerable<int> nums = list.Where(x => x > 10);
```

### 🔹 IQueryable<T>

* Executes **at data source (e.g., database)**

* Query is translated into SQL (when using EF Core)

### Example:

```csharp id="i02"
IQueryable<User> users = dbContext.Users.Where(u => u.Age > 18);
```

* Filter happens in SQL before data is returned

### Key Difference:

| Feature     | IEnumerable           | IQueryable                |
| ----------- | --------------------- | ------------------------- |
| Execution   | In-memory             | Database/server           |
| Performance | Slower for large data | Efficient (filters early) |
| Use case    | Local collections     | Remote data (DB, API)     |

---

### Interview One-liner:

> IEnumerable works in memory, while IQueryable executes queries at the data source like a database.

## 2. Explain "Deferred Execution" in LINQ. How do you force immediate execution?

**Answer:**

LINQ queries are **not executed when defined**, but when they are **iterated**.

---

### Example:


Execution occurs when:

```csharp id="d02"

    Console.WriteLine(n);
}
```

---

### Why deferred execution?

* Improves performance
* Avoids unnecessary computation
* Allows query composition

---

### 🔹 Immediate Execution

You can force execution using LINQ methods:

* `.ToList()`
* `.ToArray()`
* `.Count()`
* `.First()`

### Example:

```csharp id="d03"
var result = numbers.Where(x => x > 10).ToList();
```

* Executes immediately and stores result in memory

---

### Interview One-liner:

> Deferred execution means LINQ queries execute only when iterated, and immediate execution is forced using methods like ToList() or ToArray().

### Key Difference:

| Feature     | Thread              | Task                    |
| ----------- | ------------------- | ----------------------- |
| Level       | Low-level           | High-level              |
| Management  | Manual              | Managed by Thread Pool  |
| Performance | Expensive           | Lightweight             |
| Usage       | Rare in modern apps | Standard in modern .NET |

### Interview One-liner:

## 2. What happens if you use `.Wait()` or `.Result` instead of `await`? (Deadlocks)

**Answer:**

Using `.Wait()` or `.Result` **blocks the current thread** until the Task completes.

---

### Problem:

In UI or ASP.NET (classic) environments, this can cause a **deadlock**.

---

### Why deadlock happens:

1. Async method starts
2. It tries to resume on the original synchronization context (UI/Request thread)
3. But that thread is **blocked by `.Wait()` or `.Result`**
4. So continuation can never run → **deadlock**

---

### Example:

```csharp id="d01"
var result = GetDataAsync().Result; // Risk of deadlock
```

---

### Safe approach:

```csharp id="d02"
var result = await GetDataAsync();
```

---

### Key Difference:

| Approach          | Behavior     |
| ----------------- | ------------ |
| await             | Non-blocking |
| .Result / .Wait() | Blocking     |

---

### Interview One-liner:

> .Result and .Wait() block the thread and can cause deadlocks, while await avoids blocking by asynchronously resuming execution.

---

## 3. What is the purpose of `ValueTask<T>` and when should you use it over `Task<T>`?

**Answer:**

### 🔹 Task<T>

* Always allocates a **Task object**
* Safe and commonly used

---

### 🔹 ValueTask<T>

* Struct-based alternative to Task
* Avoids allocations when result is already available

---

### When ValueTask is useful:

* High-performance scenarios
* Frequently synchronous results
* Library-level optimizations

---

### Example:

```csharp id="v01"
ValueTask<int> GetValueAsync()
{
    return new ValueTask<int>(42); // no allocation
}
```

---

### ⚠️ When NOT to use ValueTask:

* If result is always async
* If you need multiple awaits
* If simplicity is preferred

---

### Key Difference:

| Feature    | Task<T>        | ValueTask<T>      |
| ---------- | -------------- | ----------------- |
| Type       | Reference type | Struct            |
| Allocation | Always         | Sometimes avoided |
| Complexity | Simple         | More complex      |

---

### Interview One-liner:

> ValueTask reduces allocations in high-performance scenarios, but Task is preferred for simplicity and most use cases.

---

## 4. How do you handle exceptions in an async method?

**Answer:**

Exceptions in async methods are handled using **try-catch blocks**, just like synchronous code.

---

### Example:

```csharp id="e01"

    var data = await GetDataAsync();
}

{
    Console.WriteLine(ex.Message);
}
```

---

### Important Points:

### 🔹 1. Exceptions are captured in Task

* If a Task fails, exception is stored inside it
* It is thrown when awaited

---

### 🔹 2. Using Task.WhenAll

If multiple tasks fail, you can inspect all exceptions:

```csharp id="e02"

    Console.WriteLine(ex.Message);
}
```

---

### 🔹 3. Unobserved exceptions

* If you never await a Task, exceptions may be lost or trigger runtime events

---

### Interview One-liner:

> Async method exceptions are handled using try-catch, and exceptions are propagated when the Task is awaited.

## 1. What is the difference between `throw` and `throw ex;`?

**Answer:**

* **Preserves the original stack trace**

```csharp
    throw;
}
```

### 🔹 `throw ex;`

* Throws the exception **again as a new exception**
* **Resets the stack trace** ❌

```csharp

{
    throw ex;
}
```

---

### 🔥 Why this matters:

* Losing stack trace makes debugging difficult
* You won’t know where the exception originally occurred

---

### ✅ Best Practice:

> Always use `throw` instead of `throw ex;` to preserve stack trace.

---

### Interview One-liner:

> `throw` preserves the original stack trace, while `throw ex;` resets it and should be avoided.

The `finally` block is used to execute code **regardless of whether an exception occurs or not**.

* Cleaning up operations

```csharp
}
```

### Key Points:

### ⚠️ Note:

### Interview One-liner:

# ASP.NET Core Middleware Pipeline – Interview Q&A

## 1. What is Middleware, and how does it differ from Request Filters?

**Answer:**

### 🔹 Middleware

Middleware is a **software component that sits in the HTTP request pipeline** and processes requests and responses.

Each middleware can:

* Handle request
* Modify request/response
* Pass control to next middleware

---

### 🔹 Request Filters (MVC Filters)

Filters work **inside ASP.NET Core MVC pipeline**, after routing selects a controller.

Types:

* Authorization filters
* Action filters
* Exception filters
* Result filters

---

### Key Difference:

| Feature  | Middleware                      | Filters                    |
| -------- | ------------------------------- | -------------------------- |
| Scope    | Entire request pipeline         | MVC only                   |
| Runs     | Before routing / after response | Inside controller pipeline |
| Use case | Logging, auth, error handling   | Action-level logic         |

---

### Interview One-liner:

> Middleware works across the entire HTTP pipeline, while filters operate only within the MVC action execution pipeline.

## 2. What is the Request Pipeline? Why does middleware order matter?

**Answer:**

The **Request Pipeline** is a sequence of middleware components that process an HTTP request in order.

---

### Flow:

```
Request → Middleware 1 → Middleware 2 → Middleware 3 → Controller → Response → Reverse path
```

---

### Why order matters:

Each middleware:

* Can **stop request processing**
* Can **modify request/response**
* Controls execution flow

---

### Example:

```csharp id="ppl01"
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();
```

---

### Wrong order example:

If `UseAuthorization()` is before `UseAuthentication()`, authorization will fail because user identity is not set.

---

### Interview One-liner:

> Middleware order is critical because each middleware executes sequentially and can short-circuit or modify the request.

---

## 3. Difference between `app.Run()`, `app.Use()`, and `app.Map()`

**Answer:**

---

### 🔹 app.Use()

* Adds middleware to pipeline
* Can call `next()` to pass control

```csharp id="u01"
app.Use(async (context, next) =>
{
    Console.WriteLine("Before");
    await next();
    Console.WriteLine("After");
});
```

---

### 🔹 app.Run()

* Terminal middleware
* Does NOT call next middleware
* Ends pipeline

```csharp id="r01"
app.Run(async context =>
{
    await context.Response.WriteAsync("Hello World");
});
```

---

### 🔹 app.Map()

* Branches pipeline based on route
* Creates sub-pipeline

```csharp id="m01"
app.Map("/admin", adminApp =>
{
    adminApp.Run(async context =>
    {
        await context.Response.WriteAsync("Admin Area");
    });
});
```

---

### Key Difference:

| Method | Behavior                |
| ------ | ----------------------- |
| Use    | Pass-through middleware |
| Run    | Terminates pipeline     |
| Map    | Branches pipeline       |

---

### Interview One-liner:

> Use builds middleware chain, Run terminates it, and Map creates route-based pipeline branches.

---

## 4. How do you create a custom Middleware class?

**Answer:**

A custom middleware is created by defining a class with:

* Constructor accepting `RequestDelegate`
* `Invoke` or `InvokeAsync` method

---

### Step 1: Create middleware class

```csharp id="c01"
public class CustomMiddleware
{
    private readonly RequestDelegate _next;

    public CustomMiddleware(RequestDelegate next)
    {
        _next = next;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        Console.WriteLine("Before request");

        await _next(context);

        Console.WriteLine("After request");
    }
}
```

---

### Step 2: Register middleware

```csharp id="c02"
app.UseMiddleware<CustomMiddleware>();
```

---

### Optional (clean approach): Extension method

```csharp id="c03"
public static class MiddlewareExtensions
{
    public static IApplicationBuilder UseCustomMiddleware(this IApplicationBuilder app)
    {
        return app.UseMiddleware<CustomMiddleware>();
    }
}
```

---

### Interview One-liner:

> Custom middleware is a class with InvokeAsync method that processes requests and is registered using UseMiddleware<T>().

```text
Request
 → Authorization Filters
 → Resource Filters
 → Model Binding

 → Action Method
```

### 1. Authorization Filters

* Run **first**
* Used for **authentication/authorization checks**


### 2. Resource Filters

* Run **before and after model binding**
* Used for:

  * Caching
  * Performance optimizations

---

### 3. Action Filters

* Run **before and after action method**
* Used for:

  * Logging
  * Validation
  * Modifying inputs/outputs

---

### 4. Exception Filters

* Run when an **exception occurs**
* Used for:

  * Global error handling

---

### 5. Result Filters

* Run **before and after result execution**
* Used for:

  * Modifying response
  * Formatting output

---

### Summary Table:

| Filter Type   | Purpose             |
| ------------- | ------------------- |
| Authorization | Security            |
| Resource      | Caching/performance |
| Action        | Before/after action |
| Exception     | Error handling      |
| Result        | Response processing |

---

### Interview One-liner:

> Filters run inside the MVC pipeline and allow execution at different stages like authorization, action execution, and result processing.

**Answer:**

REST (Representational State Transfer) is an architectural style based on **standard HTTP protocols**.

---

### 🔹 Core Principles:

1. **Stateless**

   * Each request is independent
   * Server does not store client state

2. **Client-Server Architecture**

   * Separation of concerns
   * Client handles UI, server handles data

3. **Uniform Interface**

   * Standard HTTP methods:

     * GET → Read
     * POST → Create
     * PUT → Update
     * DELETE → Remove

4. **Resource-Based**

   * Everything is a resource (e.g., `/users`, `/products`)

5. **Representation**

   * Data exchanged in formats like JSON or XML

---

### Interview One-liner:

> REST is a stateless architecture that uses HTTP methods to operate on resources via a uniform interface.

### Interview One-liner:

# EF Core – Migrations (Interview Q&A)

---

## 1. What are EF Core Migrations, and why are they necessary?

**Answer:**

**Migrations** are a way to **track and apply changes in your data model (C# classes) to the database schema**.

---

### 🔹 Why they are needed:

* Keep **database schema in sync** with code
* Version control for database changes
* Avoid manual SQL scripts
* Enable incremental updates

---

### Example workflow:

```bash id="mig01"
Add-Migration AddUserTable
Update-Database
```

---

### What happens:

* EF compares current model vs previous state
* Generates migration file
* Applies schema changes to DB

---

### Interview One-liner:

> Migrations track model changes and apply them incrementally to the database schema.

---

## 2. What do `Up()` and `Down()` methods represent?

**Answer:**

Each migration file contains two methods:

---

### 🔹 Up()

* Defines **changes to apply to the database**
* Runs when applying migration

```csharp id="mig02a"
protected override void Up(MigrationBuilder migrationBuilder)
{
    migrationBuilder.CreateTable(
        name: "Users",
        columns: table => new
        {
            Id = table.Column<int>(),
            Name = table.Column<string>()
        });
}
```

---

### 🔹 Down()

* Defines **how to revert changes**
* Runs when rolling back migration

```csharp id="mig02b"
protected override void Down(MigrationBuilder migrationBuilder)
{
    migrationBuilder.DropTable("Users");
}
```

---

### Key Idea:

* Up → Apply changes
* Down → Undo changes

---

### Interview One-liner:

> Up applies schema changes, while Down reverts them for rollback.

---

## 3. How do you apply migrations to production safely?

**Answer:**

---

### 🔴 Avoid:

```csharp id="mig03a"
context.Database.Migrate(); // Not recommended in production
```

  * Unexpected schema changes
  * Downtime
  * Lack of control

---

### ✅ Recommended approach: Generate SQL scripts

```bash id="mig03b"
dotnet ef migrations script
```

---

### Benefits:

* Review SQL before execution
* Controlled deployment
* Works with CI/CD pipelines
* DBA approval possible

---

### 🔹 Advanced:

```bash id="mig03c"
dotnet ef migrations script --idempotent
```

* Safe for multiple environments
* Applies only missing migrations

---

### Best Practice:

* Generate script → review → deploy via pipeline

---

### Interview One-liner:

> Use SQL scripts for production deployments instead of automatic migration to ensure control and safety.

---

## 4. What is a "Snapshot" file in EF Core migrations?

**Answer:**

The **Snapshot file** is a special file that stores the **current state of the model**.

### 🔹 Purpose:

### ⚠️ Important:

### Interview One-liner:

# EF Core – Loading Strategies (Lazy, Eager, Explicit)

---

## 1. What is Eager Loading? How do you use `.Include()` and `.ThenInclude()`?

**Answer:**

**Eager Loading** loads related entities **along with the main entity in a single query**.

---

### 🔹 Why use it:

* Avoid multiple database calls
* Improve performance for related data

---

### Example:

```csharp id="ld01"
var orders = context.Orders
    .Include(o => o.Customer)
    .ToList();
```

---

### 🔹 Using `.ThenInclude()` (for nested relationships):

```csharp id="ld02"
var orders = context.Orders
    .Include(o => o.Customer)
    .ThenInclude(c => c.Address)
    .ToList();
```

---

### Key Idea:

* `Include()` → first-level navigation
* `ThenInclude()` → deeper navigation

---

### Interview One-liner:

> Eager loading fetches related data in a single query using Include and ThenInclude.

---

## 2. What is Lazy Loading? What are the prerequisites?

**Answer:**

**Lazy Loading** loads related data **only when it is accessed**.

---

### 🔹 Example:

```csharp id="ld03"
var order = context.Orders.First();
var customer = order.Customer; // triggers DB query
```

---

### 🔹 Prerequisites in EF Core:

1. Install package:

```bash id="ld04"
Microsoft.EntityFrameworkCore.Proxies
```

2. Enable in DbContext:

```csharp id="ld05"
options.UseLazyLoadingProxies();
```

3. Navigation properties must be:

* `virtual`

```csharp id="ld06"
public virtual Customer Customer { get; set; }
```

---

### 🔹 Pros:

* Loads only when needed

### 🔹 Cons:

* Can cause performance issues (multiple queries)

---

### Interview One-liner:

> Lazy loading loads related data on demand and requires virtual properties and proxy support.

---

## 3. What is Explicit Loading? When to use `.Entry().Collection().Load()`?

**Answer:**

**Explicit Loading** means you **manually load related data when needed**.

---

### Example:

```csharp id="ld07"
var order = context.Orders.First();

context.Entry(order)
    .Collection(o => o.Items)
    .Load();
```

---

### 🔹 For reference:

```csharp id="ld08"
context.Entry(order)
    .Reference(o => o.Customer)
    .Load();
```

---

### 🔹 When to use:

* You want **control over when data is loaded**
* Avoid unnecessary queries
* Conditional loading

---

### Interview One-liner:

> Explicit loading allows manual control over loading related data using Entry().Collection().Load().

---

## 4. What is the "N+1 Query Problem"?

**Answer:**

The **N+1 problem** occurs when:

* One query fetches main entities (1 query)
* Then **N additional queries** fetch related data

---

### 🔹 Example:

```csharp id="ld09"
var orders = context.Orders.ToList();

foreach (var order in orders)
{
    Console.WriteLine(order.Customer.Name); // triggers query each time
}
```

---

### 🔥 Problem:

* 1 query (orders) + N queries (customers)
* Causes performance degradation

---

### 🔹 Which strategy causes it?

👉 **Lazy Loading**

---

### 🔹 Solution:

* Use **Eager Loading (`Include`)**
* Or batch queries

---

### Interview One-liner:

> The N+1 problem occurs when multiple queries are executed for related data, usually caused by lazy loading.

# SOLID Principles – Design Patterns & Principles (Interview Q&A)

---

## 1. Explain each letter of SOLID and why they matter

**Answer:**

SOLID is a set of **5 principles for writing clean, maintainable, and scalable object-oriented code**.

---

### 🔹 S – Single Responsibility Principle (SRP)

A class should have **only one reason to change**.

---

### 🔹 O – Open/Closed Principle (OCP)

Software should be **open for extension but closed for modification**.

---

### 🔹 L – Liskov Substitution Principle (LSP)

Derived classes should be **replaceable for their base class without breaking behavior**.

---

### 🔹 I – Interface Segregation Principle (ISP)

Clients should not be forced to depend on **interfaces they don’t use**.

---

### 🔹 D – Dependency Inversion Principle (DIP)

Depend on **abstractions (interfaces)**, not concrete implementations.

---

### Why important:

* Improves maintainability
* Reduces coupling
* Makes code testable
* Supports scalability

---

### Interview One-liner:

> SOLID principles help build loosely coupled, maintainable, and scalable object-oriented systems.

---

## 2. What is Single Responsibility Principle (SRP)? How does it prevent God Objects?

**Answer:**

### 🔹 SRP:

A class should have **one responsibility and one reason to change**.

---

### 🔴 Violating SRP (God Object):

```csharp
public class UserService
{
    public void SaveUser() { }
    public void SendEmail() { }
    public void GenerateReport() { }
}
```

👉 Problem:

* Too many responsibilities
* Hard to maintain and test

---

### ✅ Following SRP:

```csharp
public class UserService
{
    public void SaveUser() { }
}

public class EmailService
{
    public void SendEmail() { }
}

public class ReportService
{
    public void GenerateReport() { }
}
```

---

### Key Benefit:

* Each class is focused
* Easier to modify and test

---

### Interview One-liner:

> SRP ensures a class has a single responsibility, preventing large, hard-to-maintain God objects.

---

## 3. Open/Closed Principle (OCP) – Real-world example

**Answer:**

### 🔴 Violating OCP:

```csharp
public class DiscountCalculator
{
    public double Calculate(string type)
    {
        if (type == "Regular") return 10;
        if (type == "Premium") return 20;

        return 0;
    }
}
```

👉 Problem:

* Every new type requires modifying existing code

---

### ✅ Following OCP (using polymorphism):

```csharp
public interface IDiscount
{
    double Calculate();
}

public class RegularDiscount : IDiscount
{
    public double Calculate() => 10;
}

public class PremiumDiscount : IDiscount
{
    public double Calculate() => 20;
}
```

---

### Usage:

```csharp
public class DiscountService
{
    public double GetDiscount(IDiscount discount)
    {
        return discount.Calculate();
    }
}
```

---

### Benefit:

* Add new behavior without modifying existing code

---

### Interview One-liner:

> OCP allows extending functionality using new classes instead of modifying existing ones.

---

## 4. Liskov Substitution Principle (LSP)

**Answer:**

### 🔹 Definition:

A subclass should be **fully replaceable** for its base class without altering correctness.

---

### 🔴 Violation Example:

```csharp
public class Bird
{
    public virtual void Fly() { }
}

public class Ostrich : Bird
{
    public override void Fly()
    {
        throw new Exception("Cannot fly");
    }
}
```

👉 Problem:

* Breaks expected behavior

---

### 🔥 Issue:

* Runtime errors when substituting

---

### ✅ Fix:

```csharp
public interface IBird { }

public interface IFlyingBird
{
    void Fly();
}
```

---

### Interview One-liner:

> LSP ensures derived classes behave consistently with base classes to avoid runtime issues.

---

## 5. Interface Segregation Principle (ISP)

**Answer:**

### 🔹 Definition:

Clients should not depend on **methods they don’t use**.

---

### 🔴 Bad design:

```csharp
public interface IWorker
{
    void Work();
    void Eat();
}
```

---

### Problem:

* Robots don’t eat

---

### ✅ Good design:

```csharp
public interface IWorkable
{
    void Work();
}

public interface IFeedable
{
    void Eat();
}
```

---

### Benefit:

* Lean and focused interfaces
* Better maintainability

---

### Interview One-liner:

> ISP promotes smaller, specific interfaces instead of large, general-purpose ones.

---

## 6. Dependency Inversion vs Dependency Injection

**Answer:**

---

### 🔹 Dependency Inversion Principle (DIP)

* High-level modules should depend on **abstractions**
* Not on concrete classes

```csharp
public interface IMessageService
{
    void Send();
}
```

---

### 🔹 Dependency Injection (DI)

* Technique to **provide dependencies from outside**

```csharp
public class Notification
{
    private readonly IMessageService _service;

    public Notification(IMessageService service)
    {
        _service = service;
    }
}
```

---

### Key Difference:

| Concept | Meaning                  |
| ------- | ------------------------ |
| DIP     | Design principle         |
| DI      | Implementation technique |

---

### Relationship:

* DI helps achieve DIP

---

### Interview One-liner:

> DIP is a principle of depending on abstractions, while DI is a technique to inject those dependencies.

# Repository & Unit of Work Patterns – Interview Q&A

* Cleaner architecture

### Interview One-liner:

## 3. What is the Unit of Work (UoW) pattern?

**Answer:**

The **Unit of Work** pattern manages **multiple repository operations as a single transaction**.

### 🔹 Goal:

* Ensure **atomicity** (all succeed or all fail)

### Example:

# Creational Patterns – Singleton & Factory (Interview Q&A)

---

## 1. What is the Singleton Pattern? How to implement thread-safe in C#?

**Answer:**

The **Singleton Pattern** ensures that:

* Only **one instance** of a class exists
* Provides a **global access point**

---

### 🔹 Thread-safe Singleton (recommended – Lazy)

```csharp id="sg01"
public sealed class Singleton
{
    private static readonly Lazy<Singleton> _instance =
        new Lazy<Singleton>(() => new Singleton());

    public static Singleton Instance => _instance.Value;

    private Singleton() { }
}
```

---

### 🔹 Why this is thread-safe:

* `Lazy<T>` ensures:

  * Single instance creation
  * Thread-safe initialization
  * Lazy loading (created only when needed)

---

### 🔹 Alternative (double-check locking – older approach)

```csharp id="sg02"
private static readonly object _lock = new object();
private static Singleton _instance;

public static Singleton Instance
{
    get
    {
        if (_instance == null)
        {
            lock (_lock)
            {
                if (_instance == null)
                    _instance = new Singleton();
            }
        }
        return _instance;
    }
}
```

---

### Interview One-liner:

> Singleton ensures a single instance globally, and in C# it is best implemented using Lazy<T> for thread safety.

## 2. Simple Factory vs Factory Method vs Abstract Factory

### Interview One-liner:

# Behavioral Patterns – Strategy Pattern (Interview Q&A)

---

## 1. What is the Strategy Pattern? How does it support Open/Closed Principle?

**Answer:**

The **Strategy Pattern** defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime.

---

### 🔹 Idea:

* Move behavior into separate classes
* Select behavior dynamically

---

### Example:

```csharp id="st01"
public interface IPaymentStrategy
{
    void Pay();
}

public class CreditCardPayment : IPaymentStrategy
{
    public void Pay() => Console.WriteLine("Paid via Credit Card");
}

public class PayPalPayment : IPaymentStrategy
{
    public void Pay() => Console.WriteLine("Paid via PayPal");
}
```

---

### Usage:

```csharp id="st02"
public class PaymentContext
{
    private readonly IPaymentStrategy _strategy;

    public PaymentContext(IPaymentStrategy strategy)
    {
        _strategy = strategy;
    }

    public void Execute() => _strategy.Pay();
}
```

---

### 🔹 How it supports OCP:

* You can **add new strategies without modifying existing code**
* Just create a new class implementing the interface

---

### Interview One-liner:

> Strategy pattern encapsulates algorithms into separate classes, allowing behavior to be extended without modifying existing code.

---

## 2. Strategy vs if/else or switch

**Answer:**

---

### 🔴 Using if/else:

```csharp id="st03"
public void Pay(string type)
{
    if (type == "Card") { }
    else if (type == "PayPal") { }
}
```

---

### Problems:

* Violates Open/Closed Principle
* Hard to maintain
* Grows messy with more conditions

---

### ✅ Using Strategy:

* Each logic is isolated
* No conditional branching
* Easier to extend

---

### When is Strategy justified?

---

### ✅ Use Strategy when:

* Multiple algorithms exist
* Behavior changes at runtime
* Code has many conditionals
* You want clean separation

---

### ❌ Avoid when:

* Only 2–3 simple conditions
* Over-engineering for small logic

---

### Interview One-liner:

> Strategy replaces complex conditionals with polymorphism, making the code cleaner and extensible.

---

## 3. Strategy with IoC Container (Dynamic Resolution in .NET)

**Answer:**

You can use **Dependency Injection (DI)** to resolve strategies dynamically.

---

### Step 1: Register all strategies

```csharp id="st04"
services.AddScoped<IPaymentStrategy, CreditCardPayment>();
services.AddScoped<IPaymentStrategy, PayPalPayment>();
```

---

### Step 2: Inject all strategies

```csharp id="st05"
public class PaymentService
{
    private readonly IEnumerable<IPaymentStrategy> _strategies;

    public PaymentService(IEnumerable<IPaymentStrategy> strategies)
    {
        _strategies = strategies;
    }

    public void Process(string type)
    {
        var strategy = _strategies.First(s => s.GetType().Name.Contains(type));
        strategy.Pay();
    }
}
```

---

### 🔹 Better approach (using dictionary)

```csharp id="st06"
private readonly Dictionary<string, IPaymentStrategy> _map;

public PaymentService(IEnumerable<IPaymentStrategy> strategies)
{
    _map = strategies.ToDictionary(s => s.GetType().Name);
}
```

---

### 🔹 Advanced (Factory + DI):

* Combine Strategy + Factory for cleaner selection

---

### Benefits:

* Fully dynamic behavior
* No switch/if
* Easy to extend

---

### Interview One-liner:

> In .NET, Strategy can be implemented dynamically using DI by injecting multiple implementations and selecting the appropriate one at runtime.

# Web & API Concepts – RESTful API Design (Interview Q&A)

## 1. What are the six architectural constraints of REST?

**Answer:**

REST (Representational State Transfer) defines **six constraints** that ensure scalable and consistent web APIs.

---

### 🔹 1. Client-Server Architecture

* Separation of concerns
* Client handles UI, server handles data/business logic

---

### 🔹 2. Statelessness

* Each request is independent
* No session stored on server

---

### 🔹 3. Cacheability

* Responses must define whether they can be cached
* Improves performance

---

### 🔹 4. Uniform Interface

* Standardized interaction using HTTP methods:

  * GET, POST, PUT, DELETE

---

### 🔹 5. Layered System

* Architecture can have multiple layers (API gateway, load balancer, etc.)
* Client is unaware of layers

---

### 🔹 6. Code on Demand (optional)

* Server can send executable code (rarely used)

---

### Interview One-liner:

> REST defines six constraints including statelessness, uniform interface, and layered architecture to ensure scalability and simplicity.

---

## 2. What does Stateless mean in REST?

**Answer:**

A **stateless API** means:

> The server does NOT store any client session data between requests.

---

### 🔹 Each request must contain:

* Authentication info
* Required data
* Context for processing

---

### Example:


### 🔴 Not allowed:

* Server remembering user session state

---

### 🔹 Benefits:

* Scalability
* Easier load balancing
* Fault tolerance

---

### Interview One-liner:

> Stateless means every request is independent and contains all information needed for processing.

# JWT (JSON Web Tokens) – Interview Q&A

---

## 1. What are the three parts of a JWT?

**Answer:**

A **JWT (JSON Web Token)** has **three parts separated by dots (`.`)**:

```
Header.Payload.Signature
```

---

### 🔹 1. Header

Contains metadata about the token:

* Type (JWT)
* Signing algorithm (e.g., HMAC SHA256, RSA)

```json id="jwt01a"
{
  "alg": "HS256",
  "typ": "JWT"
}
```

---

### 🔹 2. Payload (Claims)

Contains **user data (claims)**:

* User ID
* Roles
* Expiry time
* Custom data

```json id="jwt01b"
{
  "sub": "123",
  "name": "John",
  "role": "Admin",
  "exp": 1710000000
}
```

---

### 🔹 3. Signature

Used to **verify integrity and authenticity**

```
HMACSHA256(
  base64(header) + "." + base64(payload),
  secret_key
)
```

---

### Interview One-liner:

> A JWT consists of Header (metadata), Payload (claims), and Signature (security verification).

---

## 2. How do you secure a JWT from tampering?

**Answer:**

JWT security is ensured using the **Signature part**.

---

### 🔹 How it works:

* Token is signed using:

  * Secret key (HMAC)
  * OR public/private key (RSA)

---

### 🔹 Verification process:

1. Server recomputes signature
2. Compares with received signature
3. If mismatch → token is invalid

---

### 🔴 Important:

* Payload is NOT encrypted (only encoded)
* Anyone can decode it
* But cannot modify without breaking signature

---

### 🔹 Extra security measures:

* Use HTTPS
* Set short expiration (`exp`)
* Rotate signing keys
* Validate issuer and audience

---

### Interview One-liner:

> JWT integrity is protected using cryptographic signatures, ensuring any tampering invalidates the token.

---

## 3. Where should JWT be stored? Local Storage vs HttpOnly Cookies

**Answer:**

---

## 🔹 1. Local Storage

```js id="jwt02a"
localStorage.setItem("token", jwt);
```

### ✔ Pros:

* Easy to use
* Accessible via JavaScript

### ❌ Cons:

* Token can be stolen via malicious scripts

---

## 🔹 2. HttpOnly Cookies

```http id="jwt02b"
Set-Cookie: token=jwt; HttpOnly; Secure
```

### ✔ Pros:

* Protected from JavaScript access
* Safer against XSS

### ❌ Cons:

* Vulnerable to CSRF (if not protected)
* Less flexible for APIs

---

## 🔹 Best Practice:

| Scenario    | Recommendation               |
| ----------- | ---------------------------- |
| SPAs        | HttpOnly Cookies (preferred) |
| Simple apps | Local Storage (less secure)  |

---

### Interview One-liner:

> HttpOnly cookies are safer than local storage because they prevent JavaScript access and reduce XSS risks.

## 4. What is a Refresh Token and why is it used?

**Answer:**

---

### 🔹 Access Token:

* Short-lived (15 min – 1 hr)
* Used for API calls

---

### 🔹 Refresh Token:

* Long-lived (days/months)
* Used to generate new access tokens

---

### 🔹 Flow:

1. User logs in → gets:

   * Access Token
   * Refresh Token

2. Access token expires

3. Client sends refresh token:

```http id="jwt03"
POST /refresh-token
```

4. Server issues new access token

---

### 🔹 Why it is used:

* Improves security (short-lived access tokens)
* Avoids frequent login
* Reduces risk if token is stolen

---

### 🔴 Important:

* Refresh tokens must be stored securely
* Usually stored in HttpOnly cookies

---

### Interview One-liner:

> Refresh tokens are used to obtain new access tokens without requiring the user to log in again, improving security and user experience.

## 2. OAuth 2.0 vs OpenID Connect (OIDC)

**Answer:**

* Authorization framework
* Focus: **Access to resources**
* Issues **access tokens**

✔ Example:

* "Allow this app to access your Google Drive"

---

### 🔹 OpenID Connect (OIDC)

* Built on top of OAuth 2.0
* Focus: **Authentication (who you are)**
* Issues **ID token + access token**

✔ Example:

* "Login with Google"

---

### Key Difference:

| Feature       | OAuth 2.0     | OpenID Connect          |
| ------------- | ------------- | ----------------------- |
| Purpose       | Authorization | Authentication          |
| Token type    | Access token  | ID token + Access token |
| Identity info | Not provided  | Provided                |

### Interview One-liner:

> OAuth 2.0 is for authorization, while OpenID Connect is for authentication built on top of OAuth.

## 3. What is Authorization Code Flow? Why is it more secure?

**Answer:**

---

### 🔹 Authorization Code Flow:

1. User logs in via Authorization Server
2. Server returns **authorization code**
3. Client exchanges code for **access token**

---

### 🔹 Flow:

```text id="oauth01a"
User → Login → Authorization Code → Client → Token Exchange → Access Token
```

---

### 🔹 Why it is secure:

* Tokens are NOT exposed in browser URL
* Uses **server-side exchange**
* Supports client authentication
* Prevents token interception

---

### 🔴 Implicit Flow (older approach):

* Tokens returned directly in browser URL
* Vulnerable to token leakage

---

### Key Difference:

| Flow               | Security         |
| ------------------ | ---------------- |
| Authorization Code | High             |
| Implicit           | Low (deprecated) |

---

### Interview One-liner:

> Authorization Code Flow is more secure because tokens are exchanged server-side instead of being exposed in the browser.

# CORS (Cross-Origin Resource Sharing) – Interview Q&A

---

## 1. What is CORS, and why do browsers enforce it?

**Answer:**

### 🔹 CORS (Cross-Origin Resource Sharing)

CORS is a **browser security mechanism** that controls how a web page can request resources from a different origin (domain, port, or protocol).

---

### 🔹 Example of cross-origin:

* Frontend: `https://app.com`
* Backend: `https://api.com`

---

### 🔹 Why browsers enforce it:

* Prevents **malicious websites** from reading sensitive data from another site
* Protects users from **Cross-Site attacks**
* Enforced by the **browser, not the server**

---

### 🔴 Important:

* Server still receives the request
* Browser blocks the response if CORS policy fails

---

### Interview One-liner:

> CORS is a browser security feature that restricts cross-origin requests to prevent unauthorized access to resources.

---

## 2. What is a Preflight Request (OPTIONS)? When is it triggered?

**Answer:**

### 🔹 Preflight Request:

A **preliminary request using HTTP OPTIONS** sent by the browser to check if the actual request is safe to send.

---

### 🔹 When it is triggered:

Browser sends a preflight request when:

* Using non-simple HTTP methods:

  * PUT, DELETE, PATCH
* Using custom headers:

  * `Authorization`
* Sending JSON with content-type:

  * `application/json`

---

### 🔹 Flow:

```text id="cors01a"
OPTIONS /api/data  → Preflight check
POST /api/data     → Actual request
```

---

### 🔹 Purpose:

* Verify server allows the request
* Check allowed methods and headers

---

### Interview One-liner:

> A preflight request is an OPTIONS call used by browsers to verify whether a cross-origin request is safe to execute.

---

## 3. How to configure CORS in ASP.NET Core?

**Answer:**

---

### 🔹 Step 1: Add CORS policy

```csharp id="cors02a"
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowFrontend",
        policy =>
        {
            policy.WithOrigins("https://frontend.com")
                  .AllowAnyHeader()
                  .AllowAnyMethod();
        });
});
```

---

### 🔹 Step 2: Enable middleware

```csharp id="cors02b"
app.UseCors("AllowFrontend");
```

---

### 🔹 Options:

| Method         | Meaning                |
| -------------- | ---------------------- |
| WithOrigins    | Allow specific domains |
| AllowAnyHeader | Allow all headers      |
| AllowAnyMethod | Allow GET/POST/etc.    |

---

### 🔹 Best practice:

* Never use `AllowAnyOrigin()` in production APIs

---

### Interview One-liner:

> In ASP.NET Core, CORS is configured using policies that specify allowed origins, headers, and methods.

---

## 4. Does CORS protect APIs from Postman or cURL?

**Answer:**

### 🔴 No, it does NOT protect against Postman or cURL.

---

### 🔹 Why?

* CORS is enforced by the **browser only**
* Postman and cURL:

  * Do NOT follow browser CORS rules
  * Can directly call APIs

---

### 🔹 What CORS actually protects:

* Only browser-based JavaScript requests
* Prevents unauthorized cross-origin reads

---

### 🔹 What it does NOT protect:

* Direct API access
* Server-to-server calls
* Postman/cURL requests

---

### 🔥 Key Insight:

CORS is a **browser security feature, not an API security mechanism**

---

### Interview One-liner:

> CORS only protects browser-based requests and does not prevent tools like Postman or cURL from accessing the API.
